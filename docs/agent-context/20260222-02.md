# [TICKET-20260222-02] `inkStorage` 壊れた保存データ耐性強化

## 目的 / 対象読者（coding agent）
`localStorage` の `ink` データが破損・改ざんされていても描画処理がクラッシュしないようにし、アプリの可用性を高める。主対象はストレージ層と Ink 描画層を実装するエージェント。

## 作成日
2026-02-22

## 優先度 / 状態
- 優先度: High
- status: `TODO`

## 背景と問題
1. `loadInkState` は配列かどうかのみ検証し、stroke/point 構造の妥当性を保証していない。
- 根拠: `src/lib/inkStorage.ts:19`
- 根拠: `src/lib/inkStorage.ts:21`

2. 描画側は `stroke.points.length` 等に直接アクセスするため、不正構造が混入すると実行時エラーの可能性がある。
- 根拠: `src/components/InkOverlay.tsx:51`
- 根拠: `src/components/InkOverlay.tsx:52`
- 根拠: `src/components/InkOverlay.tsx:65`

3. テストは「JSON破損」のみで、JSONは妥当だがスキーマが壊れているケースを網羅していない。
- 根拠: `src/lib/__tests__/inkStorage.test.ts:36`

## 実装方針
1. `inkStorage` にスキーマ検証を追加する。
- `isValidPoint`: `x`,`y` が number かつ `0..1` 範囲。
- `isValidStroke`: `points` が有効 point 配列、`color` が string、`width` が正の有限数、`ts` が有限数。
- ブロック配列内は `filter` で有効 stroke のみ採用する。

2. 正常データ互換を維持する。
- `createEmptyInkState()` の形は維持し、無効値は空配列へフォールバックする。
- ストレージキーと保存フォーマットのトップレベル構造は変更しない。

3. テストを拡張する。
- `src/lib/__tests__/inkStorage.test.ts` に以下を追加:
  - 不正 point（文字列座標、範囲外座標）
  - 不正 stroke（points 非配列、width 不正、color 不正）
  - 有効/無効混在時に有効分だけ復元されるケース

## 受け入れ基準
1. 不正スキーマの `ink` データでもアプリ起動時にクラッシュしない。
2. 正常スキーマの `ink` データは既存どおり復元される。
3. `src/lib/__tests__/inkStorage.test.ts` にスキーマ不正ケースのテストが追加され、全件成功する。
4. `src/components/__tests__/InkOverlay.test.tsx` が回帰なく成功する。

## 検証手順
1. `npm run test -- src/lib/__tests__/inkStorage.test.ts`
2. `npm run test -- src/components/__tests__/InkOverlay.test.tsx`
3. `npm run test`
