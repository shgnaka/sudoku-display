# [TICKET-20260222-03] `SideDrawer` フォーカス/アクセシビリティ改善

## 目的 / 対象読者（coding agent）
閉状態のドロワーにキーボードフォーカスが入る不整合を解消し、ナビゲーション操作のアクセシビリティを改善する。主対象はナビゲーションUIを実装するエージェント。

## 作成日
2026-02-22

## 優先度 / 状態
- 優先度: Medium
- status: `TODO`

## 背景と問題
1. `SideDrawer` は閉状態でもDOM上に存在し、視覚的に画面外へ移動する実装になっている。
- 根拠: `src/components/navigation/SideDrawer.tsx:11`
- 根拠: `src/components/navigation/SideDrawer.tsx:15`
- 根拠: `src/styles/app-shell.css:115`

2. 閉状態で非表示扱いの明示（`aria-hidden`/`inert` 等）がなく、Tab移動時のフォーカス侵入リスクがある。
- 根拠: `src/components/navigation/SideDrawer.tsx:15`
- 根拠: `src/components/navigation/SideDrawer.tsx:23`

## 実装方針
1. 採用案は「条件レンダリング + フォーカス復帰」。
- `isOpen === false` のとき `drawer-backdrop` と `aside` をレンダリングしない。
- `isOpen === true` のときのみドロワーDOMをマウントする。

2. 開閉時フォーカス制御を追加する。
- `AppHeader` のメニューボタン要素を `ref` で保持する。
- ドロワーを閉じた際は、メニューボタンへフォーカスを戻す。
- ドロワーを開いた直後は、最初のナビゲーションボタンへフォーカス移動する。

3. A11y属性を補完する。
- `aside` に `aria-modal="true"` と適切なラベル付けを追加する。
- 背景クリック・Escape 閉じる動作は維持する。

4. テストを新規追加する。
- `src/components/navigation/__tests__/SideDrawer.test.tsx` を作成し、以下を検証:
  - 閉状態でドロワー要素がDOMに存在しない。
  - 開閉時のフォーカス遷移が正しい。
  - ルート遷移後にメニューボタンへ戻る。

## 受け入れ基準
1. 閉状態でTab移動してもドロワー内要素へフォーカスが入らない。
2. 開状態で最初のメニュー項目へフォーカスできる。
3. 閉じる操作後にハンバーガーボタンへフォーカスが戻る。
4. 既存の遷移動作（ボタン選択、Escape閉じる、背景クリック閉じる）を維持する。

## 検証手順
1. `npm run test -- src/components/navigation/__tests__/SideDrawer.test.tsx`
2. `npm run test`

## 非目標（今回やらないこと）

## リスク / ロールバック
- リスク: 条件レンダリング化で開閉アニメーションの見え方が変わる可能性がある。
- ロールバック: アニメーション要件が満たせない場合、`inert`/`aria-hidden` 方式に切り替える。
