# [TICKET-20260222-04] GitHub Actions で TS/Rust テストを必須化

## 目的 / 対象読者（coding agent）
デプロイ前に TypeScript と Rust のテストをCIで必須化し、回帰を早期検出する。主対象はCI設定と品質ゲートを運用するエージェント。

## 作成日
2026-02-22

## 優先度 / 状態
- 優先度: Medium
- status: `TODO`

## 背景と問題
1. 現行ワークフローはスタイルlintとbuildのみで、テスト未実行のままデプロイ可能。
- 根拠: `.github/workflows/deploy-pages.yml:42`
- 根拠: `.github/workflows/deploy-pages.yml:45`

2. アプリ側には `npm run test` が定義されているが、CIで利用されていない。
- 根拠: `package.json:15`

3. Rust側ユニットテストが存在するが、CIで検証されていない。
- 根拠: `rust/sudoku_generator/src/lib.rs:279`
- 根拠: `rust/sudoku_generator/src/lib.rs:300`

## 実装方針
1. `deploy-pages.yml` の `build` ジョブにテストステップを追加する。
- 追加順序:
  1. `npm run lint:styles`
  2. `npm run test`
  3. `cargo test --manifest-path rust/sudoku_generator/Cargo.toml`
  4. `npm run build`

2. 失敗時挙動を明示する。
- いずれかのステップ失敗で `build` ジョブを失敗させ、`deploy` ジョブへ進ませない。

3. 実行時間増加を許容する。
- 品質優先でテスト必須化を採用し、必要なら後続でジョブ分割を検討する。

## 受け入れ基準
1. TSテストが失敗した場合、デプロイが停止する。
2. Rustテストが失敗した場合、デプロイが停止する。
3. すべて成功した場合のみ、既存どおり build と deploy が実行される。

## 検証手順
1. ローカルで事前確認:
- `npm run test`
- `cargo test --manifest-path rust/sudoku_generator/Cargo.toml`

2. CI確認:
- テスト失敗を含むコミットでワークフローが `build` で停止することを確認する。
- テスト成功コミットで `deploy` まで進むことを確認する。

## 非目標（今回やらないこと）
- ワークフローのマトリクス化
- PR専用ワークフローの新設

## リスク / ロールバック
- リスク: CI時間が増加し、デプロイ所要時間が伸びる。
- ロールバック: 緊急時はテストステップを一時的に外すのではなく、失敗原因を先に修正する運用を優先する。
