# App CSS Maintainability Audit

## 目的 / 対象読者（coding agent）
`src/App.css` の改善余地を、実装時に判断迷いが出ない粒度で共有する。主対象は、UI改善を担当するコーディングエージェント。

## 調査日
2026-02-22

## 現状要約
- `src/App.css` に、アプリ共通・Solveページ専用・ナビゲーション・モバイル調整が混在している。
- `solve-no-scroll` 向けの条件付きスタイルが複数箇所へ分散しており、影響範囲把握が難しい。
- 色・サイズ・境界線の値にハードコードが多く、再利用性と変更耐性が低い。
- ボタンやフォーカス表現が複数パターンで管理されており、一貫性に改善余地がある。

## 改善余地（優先度付き、ファイル:行番号付き）

### High
1. CSS責務分離の不足
- 問題: アプリ共通とSolve専用スタイルが同ファイルで混在し、改修時の影響範囲が大きい。
- 根拠: `src/App.css:49`, `src/App.css:327`, `src/App.css:381`, `src/App.css:496`
- 期待効果: 変更リスクを局所化でき、改善サイクルを短縮できる。

2. `solve-no-scroll` スタイルの散在
- 問題: `.app-root.solve-no-scroll` 系セレクタが分散し、レイアウトバグの原因追跡が難しい。
- 根拠: `src/App.css:49`, `src/App.css:159`, `src/App.css:173`, `src/App.css:333`, `src/App.css:381`
- 期待効果: 挙動確認と微調整の工数を削減できる。

3. トークン不足によるハードコード増加
- 問題: 同系統の色・半径・ボーダー値が個別記述され、統一変更に弱い。
- 根拠: `src/App.css:113`, `src/App.css:250`, `src/App.css:259`, `src/App.css:316`, `src/App.css:443`
- 期待効果: デザイン一括調整が容易になり、見た目の一貫性が上がる。

### Medium
1. ボタン系スタイルの重複
- 問題: 汎用ボタン、モードボタン、パッドボタン、ナビボタンで類似定義が分散している。
- 根拠: `src/App.css:258`, `src/App.css:288`, `src/App.css:523`, `src/App.css:685`
- 期待効果: コンポーネント間の見た目差異を意図的に管理しやすくなる。

2. フォーカス表現の適用範囲が限定的
- 問題: `.sudoku-cell:focus` はあるが、他操作要素へのフォーカス可視性が統一されていない。
- 根拠: `src/App.css:484`
- 期待効果: キーボード操作時のアクセシビリティと操作確信性が向上する。

3. レスポンシブ条件の説明不足
- 問題: モバイル用調整が1つのメディアクエリに寄っており、solve専用計算との関係が見えにくい。
- 根拠: `src/App.css:699`, `src/App.css:712`
- 期待効果: 画面幅別の仕様理解が容易になり、改修時の破綻を防ぎやすい。

### Low
1. タイポグラフィ規則の分散
- 問題: フォントサイズ・行間が各所で個別調整されている。
- 根拠: `src/App.css:95`, `src/App.css:193`, `src/App.css:457`, `src/App.css:527`
- 期待効果: 文字可読性を維持しながら段階的に調整しやすくなる。

2. 凡例のモバイル時完全非表示
- 問題: `solve-no-scroll` 時に凡例を非表示にしており、色の意味理解コストが上がる。
- 根拠: `src/App.css:584`
- 期待効果: 省スペースな代替表示を導入し、初見ユーザーの理解性を保てる。

## 非目標（今回やらないこと）
- Sudokuロジックや状態管理（TypeScript）の仕様変更
- ルーティングやページ構成の再設計
- 全ページの全面的なビジュアル刷新
- 新規デザインシステムの導入

## 次に実装すべき順序
1. `src/App.css` の責務マップを作成し、分割対象セクションを確定する。
2. `solve-no-scroll` 関連スタイルをSolve専用ブロックへ集約する。
3. 色・半径・高さ・余白のトークンを増設し、ハードコード値を段階置換する。
4. ボタン系スタイルを基底パターンへ整理し、重複を削減する。
5. `:focus-visible` を中心にフォーカス可視性を統一する。
6. レスポンシブ条件を整理し、必要なコメントを最小限で補強する。
7. モバイル凡例の代替表示方針を反映し、最終微調整を行う。
